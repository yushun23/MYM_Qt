# 风险登记册（Risk Register）

> 目的：标注旧系统"不能直接照抄"的部分，确保新架构不重蹈覆辙。

## 严重风险（必须在新架构中修复）

### R1: UI 层直接执行 SQL
- **旧代码位置**: `view_dashboard.py` 函数 `build_dashboard()`、`view_accounts.py` 函数 `load_accounts()`、`view_budget.py` 多处
- **风险**: UI 与数据库紧耦合，难以测试、难以复用、难以修改数据模型
- **新架构处理**: UI 层只能通过 Repository/QueryService 访问数据；所有查询封装在 infrastructure 层

### R2: 跨线程共享 SQLite 连接
- **旧代码位置**: `stock_*.py` 后台线程、`ai_service.py` 异步调用
- **风险**: SQLite 不支持高并发写入，跨线程共享连接导致 database locked 错误
- **新架构处理**: 每个线程/后台任务创建独立短生命周期 Session；WAL 模式 + 写入序列化

### R3: 浮点金额计算
- **旧代码位置**: 多处使用 Python float 进行金额计算
- **风险**: 浮点精度问题导致金额对不上（如 0.1 + 0.2 != 0.3）
- **新架构处理**: 全部金额使用 Decimal / Numeric(18,2)；数据库存储为 INTEGER（分）或 NUMERIC

### R4: 破坏性 Schema 迁移
- **旧代码位置**: `migrations.py` 直接在用户 .mym 文件上执行 ALTER TABLE
- **风险**: 迁移失败可能损坏用户数据，无法回滚
- **新架构处理**: Alembic 可追踪迁移 + 迁移前自动备份 + 失败不损坏已有账本

### R5: 股票资产重复统计
- **旧代码位置**: `view_dashboard.py` 仪表盘总资产计算
- **风险**: 股票联动账户和股票资金池可能被重复计入总资产
- **新架构处理**: 明确计算口径，股票资产由投资域独立统计，核心账户排除 investment_linked 类型

### R6: 缺乏审计日志
- **旧代码位置**: 缺失
- **风险**: 无法追溯谁在何时做了什么操作
- **新架构处理**: 所有资金相关写入必须写入 AuditLog

### R7: 无统一写入入口
- **旧代码位置**: AI、导入、应收、股票月结各有独立写入路径
- **风险**: 业务规则不一致，容易出现"绕开校验"的写入
- **新架构处理**: 所有写入走 P5 核心账务服务（CreateTransactionUseCase）

### R8: 历史估值无数据源说明
- **旧代码位置**: `stock_*.py` 行情查询
- **风险**: 缺行情时给出虚假精确值
- **新架构处理**: 历史估值明确标注数据来源，缺数据时显示"数据不足"

## 中等风险

### R9: 旧.mym 兼容性
- **旧代码位置**: `migrations.py`、`database.py` 兼容多种旧 schema 版本
- **风险**: 旧账本版本过多，迁移逻辑复杂
- **新架构处理**: P34-P37 专项处理旧库迁移，只读扫描 + 报告 + 受控迁移

### R10: 预算关闭后修改
- **旧代码位置**: `view_budget.py`
- **风险**: 已关闭月份的无提示修改导致历史数据不一致
- **新架构处理**: 关闭月份不可修改；修改需先重新打开，记录操作

### R11: 应收账款与普通流水的区分
- **旧代码位置**: `view_receivable.py`、`view_record.py`
- **风险**: 应收操作可能在普通流水页面被误编辑/删除
- **新架构处理**: 应收交易在普通流水中心标记清晰来源，编辑需跳转对应页面

### R12: AI 直接执行高风险操作
- **旧代码位置**: `ai_action_spec.py`、`ai_service.py`
- **风险**: AI 可能误操作或越权
- **新架构处理**: 所有 AI 动作需用户确认 + 统一 validator 校验

## 低风险

### R13: 插件加载错误
- **旧代码位置**: `plugin_loader.py`
- **风险**: 插件异常可能拖垮主程序
- **新架构处理**: 插件隔离加载，错误不影响主程序

### R14: 图表 CDN 依赖
- **旧代码位置**: `pyecharts_chart_service.py` HTML 模板
- **风险**: 离线环境图表无法加载
- **新架构处理**: 本地 ECharts 资源，不依赖 CDN

### R15: HTML 注入风险
- **旧代码位置**: 图表/报表 HTML 拼接
- **风险**: 用户输入（备注、账户名）可能注入恶意脚本
- **新架构处理**: Python 端转义所有用户输入后再传入 HTML

---

## 风险缓解策略总结

| 风险编号 | 严重程度 | 缓解措施 | 验证方式 |
|---|---|---|---|
| R1 | 严重 | 分层架构强制约束 | 代码审查 + 架构测试 |
| R2 | 严重 | Session工厂 + WAL | 并发测试 |
| R3 | 严重 | Decimal全部替换 | 金额精度测试 |
| R4 | 严重 | 备份 + Alembic | 迁移测试 |
| R5 | 严重 | 明确统计口径 | 口径回归测试 |
| R6 | 严重 | AuditLog | 审计日志测试 |
| R7 | 严重 | 统一写入服务 | 写入路径测试 |
| R8 | 中等 | 数据源标注 | 行情边界测试 |
| R9-R15 | 中/低 | 各专项处理 | 专项测试 |
