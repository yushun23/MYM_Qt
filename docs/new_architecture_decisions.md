# 新架构决策记录（Architecture Decision Records）

> 目的：记录新系统关键技术选型的原因、权衡和约束。

## ADR-001: 选择 PySide6 Widgets 而非 Flet

**状态**: 已确定

**背景**: 旧项目使用 Flet（基于 Flutter），存在以下问题：
- Flet 对桌面原生控件支持有限
- 打包体积大
- 中文输入和系统集成体验差
- 社区生态不如 Qt 成熟

**决策**: 使用 PySide6 (Qt for Python) Widgets 模式

**后果**:
- ✅ 原生桌面体验，更好的系统集成
- ✅ 成熟的控件库和文档
- ✅ 丰富的第三方资源（QWebEngineView、QWebChannel）
- ❌ 学习曲线稍高
- ❌ 需手动管理 Widget 生命周期

---

## ADR-002: 选择 SQLAlchemy 2.x + SQLite

**状态**: 已确定

**背景**: 旧项目使用手写 SQL + sqlite3，存在 SQL 注入风险、难以测试、迁移困难等问题。

**决策**: 使用 SQLAlchemy 2.x ORM + SQLite

**后果**:
- ✅ 参数化查询防注入
- ✅ ORM 简化数据操作
- ✅ Alembic 提供可追踪迁移
- ✅ Session 管理事务边界
- ❌ ORM 有一定性能开销（可接受的）

---

## ADR-003: 选择 QWebEngineView + 本地 ECharts

**状态**: 已确定

**背景**: 需要交互式图表（点击钻取、缩放等），旧项目使用 pyecharts 生成静态 HTML。

**决策**: 优先使用 QWebEngineView 加载本地 ECharts；pyecharts 仅作辅助 option 生成

**后果**:
- ✅ 交互式图表（点击、缩放、数据钻取）
- ✅ 通过 QWebChannel 双向通信
- ✅ 本地资源，不依赖 CDN
- ❌ QWebEngineView 占用内存较大
- ❌ pyecharts 不能完全依赖

---

## ADR-004: 分层架构（DDD 风格）

**状态**: 已确定

**决策**: 采用四层架构：

```
src/mym/
├── ui/              # 表现层：PySide6 Widgets
├── application/     # 应用层：Use Cases / Services / DTOs
├── domain/          # 领域层：实体 / 值对象 / 枚举 / 规则
└── infrastructure/  # 基础设施层：Repository / DB / 外部服务
```

**后果**:
- ✅ 关注点分离，每层可独立测试
- ✅ UI 可替换而不影响业务逻辑
- ❌ 初期开发速度稍慢
- ❌ 需要更多文件和目录

---

## ADR-005: 统一账务写入入口

**状态**: 已确定

**背景**: 旧系统 AI、导入、应收、股票月结各有独立写入路径，容易绕过校验。

**决策**: 所有资金写入必须通过 `CreateTransactionUseCase`(P5)

**后果**:
- ✅ 业务规则一致
- ✅ 审计日志不遗漏
- ✅ 余额重算统一触发
- ❌ 所有模块依赖核心服务

---

## ADR-006: 账户余额为可重算缓存

**状态**: 已确定

**决策**: current_balance 是缓存字段，可由 opening_balance + 已过账交易重算

**后果**:
- ✅ 数据一致性可验证
- ✅ 异常时可自动修复
- ❌ 每次查询可能需重算（用缓存优化）

---

## ADR-007: 金额使用 Decimal

**状态**: 已确定

**决策**: 所有金额使用 Python Decimal，数据库 Numeric(18,2)

**后果**:
- ✅ 精度保证
- ❌ 性能略低于 float（可接受）

---

## ADR-008: 股票模块独立可撤销

**状态**: 已确定

**背景**: 用户明确要求股票模块可隐藏、归档、按批次回滚、永久删除前备份。

**决策**:
- 股票模块有独立数据表，不混入核心流水表
- 所有导入带 ImportJob，支持按批次回滚
- 模块状态: enabled / hidden / archived
- 永久删除需备份 + 二次确认

**后果**:
- ✅ 满足产品需求
- ❌ 架构复杂度增加

---

## ADR-009: 多语言使用 JSON 资源

**状态**: 已确定

**决策**: 使用 JSON 文件管理翻译（zh-CN、en），通过 tr_key() 服务获取

**后果**:
- ✅ 易于编辑和维护
- ✅ 不依赖 Qt 翻译工具链
- ❌ 需自行管理回退逻辑

---

## ADR-010: 旧功能首期兼容策略

**状态**: 已确定

**决策**:
- **首期完整实现**: 普通记账、流水、账户、分类、仪表盘、收支报表、设置
- **次期实现**: 应收、预算、插件
- **三期实现**: 股票/投资模块
- **四期实现**: AI 助手
- **最后实现**: 旧库迁移、打包、文档

**后果**:
- ✅ 核心闭环先稳定
- ❌ 用户需等待功能完整

---

## ADR-011: 测试策略

**状态**: 已确定

**决策**: pytest + pytest-qt，分层测试
- 单元测试：domain / application 层
- 集成测试：repository + service
- UI 测试：pytest-qt
- 迁移测试：旧库样本 → 新库

**后果**:
- ✅ 可自动验证核心逻辑
- ❌ UI 测试维护成本高
